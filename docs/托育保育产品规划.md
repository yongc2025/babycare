# 托育 + 保育产品规划

## 1. 文档目的

本文档记录 BabyCare/HuiGrowth 从“家庭育儿记录型产品”升级为“真实托育 + 保育平台”的产品方向、能力边界和阶段路线。

2026-07-15 已补充用户正式需求《托育+保育平台需求（用户版-供参考）》的评估结论。本项目不照搬其完整微服务、多端、硬件和监管方案，但吸收其中对托幼一体、合规、安全、健康、祖辈端和园所运营的优秀设计。

## 2. 当前产品策略

### 架构策略

当前不拆微服务，采用“模块化单体”。

```text
一个 Spring Boot 后端
一个 React 前端
按业务模块划清边界
为未来微服务和多端拆分预留接口
```

详细决策见 `docs/ARCHITECTURE_DECISIONS.md`。

### 多端策略

当前不同时建设 PC、App、小程序、监管端。

先用一个 React Web 承载多角色工作台：

- 园长 / 机构管理员
- 老师 / 保育员
- 保健医
- 家长
- 财务
- 后勤 / 保安

后续在主链稳定后，再拆教师移动端、家长小程序、祖辈模式和监管端。

### 产品优先级

第一优先级不是硬件、AI、微服务或监管对接，而是跑通真实托育主链：

```text
机构 -> 班级 -> 入托档案 -> 考勤接送 -> 一日照护 -> 日报 -> 家长同步 -> 园所数据沉淀
```

## 3. 正式需求带来的校准

用户正式需求比原规划更完整，覆盖：

- 园所运营：招生、报名审核、试托、分班、收费、退费、后勤。
- 婴幼儿照护：喂养、睡眠、如厕、饮水、体征、活动、成长档案。
- 卫生保健：晨午检、食谱营养、传染病防控、食品留样、消毒台账。
- 安全保障：接送授权、委托接送、人脸识别、监控、事故记录、安全巡检。
- 家园共育：动态、日报、通知、亲子任务、在线约谈、家庭群组。
- 隔代共育：长辈模式、大字高对比、语音播报、接送码。
- 课程评估：0-3 岁月龄里程碑、3-6 岁五大领域评估、雷达图。
- 数据监管：园长驾驶舱、备案字段、等级评定、监管报表。
- 硬件部署：考勤机、晨检机器人、摄像头、电子班牌、环境检测等。

我们接受其产品方向，但分阶段落地。

## 4. 双龄段产品架构

正式需求中“托育端”和“保育端”的差异非常关键，后续模型和页面需要支持机构或班级级别配置。

| 维度 | 托育端 0-3 岁 | 保育端 3-6 岁 |
|---|---|---|
| 关注重点 | 精细照护、健康监测、安全防护 | 教学评估、家园互动、幼小衔接 |
| 一日流程 | 喂养、睡眠、换尿布、抚触、体征 | 进餐、午睡、活动、课程、户外 |
| 健康管理 | 发育曲线、过敏、喂养、每日体征 | 晨午检、体质测评、视力口腔筛查 |
| 评估体系 | 月龄发展里程碑 | 健康、语言、社会、科学、艺术 |
| 家园沟通 | 高频结构化日报 | 日精选、周报、月报、作品展示 |

短期先按托育主链实现，数据模型预留 `careType` / `classroomType` / `ageRange` 这类字段。

## 5. 角色与端规划

### 园长 / 机构管理员

- 机构、园区、班级、员工管理
- 招生、报名、试托、入托、分班
- 师生比、班级容量、托位使用率
- 审批、通知、数据驾驶舱
- 合规、台账、监管导出

### 老师 / 保育员

- 今日班级宝宝列表
- 到园、离园、请假、缺勤
- 晨检、全日观察
- 喂养、饮水、睡眠、如厕、体温、情绪、活动
- 照片、视频、教师评语
- 异常事件、用药执行、日报生成

### 保健医

- 晨午检管理
- 体征异常处理
- 用药审核
- 过敏标签
- 食谱营养
- 传染病防控
- 卫生保健台账导出

### 家长

- 查看到离园状态
- 查看日报、动态、照片、通知
- 请假申请
- 委托接送
- 用药申请
- 查看成长档案、体征趋势、账单

### 祖辈家长

- 大字、高对比、少入口
- 看孩子、看日报、看接送状态
- 接送码 / 实体卡
- 语音播报、语音输入

### 财务 / 运营

- 收费项目
- 账单、缴费、欠费、退款
- 收据 / 发票
- 财务报表

### 后勤 / 保安

- 接送核验
- 安全巡检
- 设施报修
- 校车 / 门禁 / 监控协作

### 监管人员

短期不做独立监管平台，先预留导出和字段清单。

## 6. 阶段路线

### L1 主链闭环

目标：让园所每天能真实使用。

- 机构、班级、员工
- 宝宝入托档案
- 考勤与接送基础
- 一日照护记录
- 结构化日报
- 家长查看

### L2 安全健康

目标：补齐托育专业性和合规可信度。

- 晨午检
- 全日观察
- 用药管理
- 过敏管理
- 异常 / 事故上报
- 接送授权、委托接送、滞留预警
- 基础安全台账

### L3 园所运营

目标：从“照护工具”升级为“园所运营系统”。

- 招生线索
- 报名审核
- 试托 / 试园
- 收费账单
- 通知公告
- 园长驾驶舱
- 师生比和班级容量预警

### L4 专业保健与教学评估

目标：吸收正式需求中的保健医和保育端能力。

- 食谱营养
- 食品留样
- 消毒台账
- 传染病防控
- 月龄发展里程碑
- 五大领域评估
- 成长时间轴

### L5 多端与智能硬件

目标：在业务闭环稳定后做平台化能力。

- 教师移动端 / PWA
- 家长小程序
- 祖辈模式
- 监管导出 / 对接
- 硬件接入抽象
- IoT 网关
- AI 日报和异常趋势识别

## 7. 当前不优先做

- 物理微服务拆分
- K8s、灰度、服务网关、链路追踪
- 原生 App / 小程序多端并行开发
- 真实监管系统对接
- 人脸识别、AI 视频分析、晨检机器人接入
- OCR、电子签、在线支付、电子发票
- 大规模 AI 包装和泛内容社区

这些能力有价值，但要在主链和真实客户场景跑通后再投入。

## 8. 下一阶段开发重点

当前应围绕正式需求校准后的 T008 推进：

```text
考勤与接送基础
```

第一版应包含：

- 幼儿到园
- 幼儿离园
- 请假申请 / 审批
- 缺勤状态
- 接送人基础信息
- 委托接送预留
- 班级某日考勤列表
- 宝宝考勤记录

后续再扩展人脸识别、临时接送码、滞留预警、黑名单等能力。

## 9. 结论

正式用户需求证明：真正的托育 + 保育平台不是简单记录宝宝成长，而是支撑园所完成“运营、照护、健康、安全、家园、监管”的完整闭环。

我们的路线应保持务实：

```text
产品上接受完整平台视野
架构上采用模块化单体
前端上采用单 Web 多角色
实现上优先跑通主链
```

这样既能贴近用户的正式需求，又能避免过早陷入微服务、多端和硬件集成的复杂度。
