# 架构决策记录

本文档记录 BabyCare/HuiGrowth 长期开发中的关键架构决策。正式用户需求中包含微服务、多端、灰度、硬件和监管等完整平台能力，但本项目按阶段落地。

## ADR-001：当前不拆微服务，采用模块化单体

状态：Accepted

日期：2026-07-15

背景：

- 用户正式需求提出微服务、API Gateway、K8s、灰度发布、RabbitMQ、IoT Gateway 等完整平台架构。
- 当前项目仍处于业务主链建设阶段，机构、班级、入托刚建立，考勤、照护、日报尚未闭环。

决策：

- 当前保留一个 Spring Boot 后端。
- 在代码和文档上按业务模块划清边界。
- 不引入服务注册、配置中心、分布式事务、链路追踪和 K8s。

当前逻辑模块边界（按数据库域前缀）：

```text
user(sys_) / organization(org_) / enrollment(enr_) / attendance(att_) / care(care_) / health(hlth_) / safety(safety_) / family(family_) / finance(fin_) / education(edu_/dev_) / baby(baby_) / hardware(hw_) / ai(ai_) / announcement(ann_) / meal(meal_) / report(无独立库，只读聚合)
```

后续拆分条件：

- 有多个团队并行维护不同业务域。
- 某个模块有独立扩缩容或部署需求。
- 硬件接入、消息通知、报表统计等出现明显独立负载。
- 单体发布频繁互相影响。

优先拆分候选：

1. hardware / iot gateway
2. notification
3. report
4. file / media
5. payment

## ADR-002：当前不拆多端，采用单 Web 多角色

状态：Accepted

日期：2026-07-15

背景：

- 用户正式需求包含园长 PC、教师 App、家长 App/小程序、祖辈小程序、保健医 PC+App、后勤 App、监管平台。
- 当前后端业务 API 尚未完整稳定，同时建设多端会造成大量返工。

决策：

- 当前保留一个 React 前端。
- 按角色拆菜单、路由和工作台。
- 优先保证 PC Web 可跑通主链。
- 教师端移动体验先用响应式页面或 PWA 思路预留。

当前角色工作台（实际路由）：

```text
/principal-workbench       园长/机构管理员
/teacher-workbench         老师/保育员
/boss-dashboard            老板/集团管理员
/system-management         系统管理员
/dashboard                 通用首页工作台
/parent-reports            家长
/elder-mode                祖辈家长
/health-safety             保健医/安全后勤
/operations-regulatory     财务/运营
/parent-applications       家长申请
/family-management         家庭成员管理
```

后续拆端条件：

- 老师端高频移动录入已经稳定。
- 家长端日报、通知、请假、接送授权已经稳定。
- 有真实用户需要微信小程序、App 推送或设备扫码。

## ADR-003：硬件能力先抽象接口，不直接做设备接入

状态：已执行（接口层已完成）

日期：2026-07-15（初始决策），2026-07-19（接口层已实现）

背景：

- 正式需求包含人脸考勤机、晨检机器人、监控、电子班牌、体征监测带、环境检测仪等硬件。
- 当前没有真实设备、协议和部署环境。

决策：

- 短期只做人工录入和可追溯记录。
- 为考勤、晨检、接送记录预留 `source`、`deviceId`、`capturedAt` 等字段。
- 不实现 MQTT、RTSP、ONVIF、GB/T28181、BLE、Modbus 等协议接入。

已执行：

- `HardwareDevice` 实体：设备档案（设备名、类型、型号、序列号、状态、绑定机构）
- `HardwareEvent` 实体：硬件原始事件沉淀（事件类型、原始数据、设备关联、处理状态）
- `HardwareIntegrationController`：设备 CRUD + 事件接收/查询
- 事件只沉淀原始事实，不自动写入考勤/健康观察/接送记录

后续触发条件：

- 拿到具体硬件型号和协议文档，开始对接真实设备

## ADR-004：监管对接先做字段和导出，不直接对接外部系统

状态：已执行（导出层已完成）

日期：2026-07-15（初始决策），2026-07-19（导出能力已实现）

背景：

- 正式需求强调托育备案、卫生保健年报、等级评定和监管上报。
- 不同地区监管系统字段、格式和接口差异较大。

决策：

- 短期先整理监管字段，保证业务数据可沉淀。
- 先做 Excel/PDF/CSV 导出。
- 不直接对接卫健委、教育局或市场监管系统。

已执行：

- 机构新增备案字段（license_number、备案号、监管等级等）
- `RegulatoryReportController`：监管信息聚合接口 + 导出端点
- 前端 `regulatoryReportAPI` + 运营监管页导出入口

后续触发条件：

- 明确目标地区，开始对接具体监管系统

## ADR-005：正式需求作为方向参考，不作为一次性全部范围

状态：执行中（L1-L4 已完成，L5 推进中）

日期：2026-07-15（初始决策），2026-07-19（阶段状态更新）

背景：

- 正式需求覆盖完整商业平台，范围包含运营、照护、健康、安全、财务、监管、硬件、多端。
- 当前项目需要先形成可验证、可迭代的产品主链。

决策：

- 正式需求中的优秀能力进入路线图。
- 任务按 L1 主链、L2 安全健康、L3 园所运营、L4 专业保健评估、L5 多端硬件平台化分阶段执行。
- 当前任何新开发都优先服务主链闭环。

当前阶段状态：

- L1-L2✅ 已完成：主链闭环 + 安全健康全部功能可用
- L3✅ 基本完成：园所运营（招生/收费/通知/驾驶舱）全部就绪
- L4✅ 基本完成：专业保健评估接近完成（食品留样/消毒台账待实现）
- L5🔄 进行中：祖辈模式/监管导出/硬件抽象/AI 对话已上线，教师移动端/家长小程序待启动

## ADR-006：优先建设真实 API，不再堆 demo 页面

状态：Accepted

日期：2026-07-15

背景：

- 现有前端曾存在较多 demo 页面和 mock fallback。
- 用户明确希望系统走向真实托育 + 保育产品。

决策：

- 新页面必须绑定真实 API 或明确展示“待接入/空状态”。
- 不新增正式 mock fallback。
- 不为了展示效果绕过后端直接写死业务数据。

## ADR-007：微服务通信与数据库关系设计原则

状态：Accepted（前瞻设计，当前单体阶段仅代码层预留）

日期：2026-07-17

> **当前实现状态说明**：本文描述的是**未来微服务拆分后的目标架构**。当前仍处于模块化单体阶段（ADR-001），以下内容（RabbitMQ、API Gateway、MinIO、Saga、CQRS、事件驱动读模型）均**尚未实现**，仅作为代码层预留的设计参考。当前跨模块调用通过 Service 层直接调用实现。

背景：

- 正式需求定义了 9 个微服务（user、operation、care、health、safety、family、education、report、iot-gateway）。
- 当前采用模块化单体（ADR-001），但需要在代码层预先划清数据边界和通信契约，为未来拆分做准备。
- T048-T080 将新增系统管理、RBAC、多园区、传染病防控等模块，需要统一的跨域数据设计原则。

决策：

### 1. 数据所有权原则

每个微服务独占写入自己的表，其他服务只能通过 API 读取：

| 服务 | 自有表范围（写权限） | 仅存关联 ID（不建物理 FK） |
|---|---|---|
| user-service | `sys_user`, `sys_role`, `sys_permission`, `sys_menu`, `sys_audit_log`, `sys_data_dict`, `sys_config` | 无 |
| operation-service | `org_*`, `enr_*`, `fin_*`, `adm_*` | `userId`, `babyId` |
| care-service | `care_*`, `dly_*`, `baby_*` | `enrollmentId` |
| health-service | `hlth_*`, `meal_*` | `enrollmentId` |
| safety-service | `att_*`, `safety_*` | `enrollmentId`, `staffId` |
| family-service | `family_*`, `ann_*` | `userId`, `enrollmentId` |
| education-service | `dev_*`, `edu_*` | `enrollmentId` |
| iot-gateway | `hw_*` | 无（只写设备事件，不依赖业务 ID） |
| report-service | 不落库（只读聚合） | 无 |

### 2. 服务间通信方式

| 通信场景 | 方式 | 协议 | 说明 |
|---|---|---|---|
| 实时读查询 | REST 同步调用 | HTTP/JSON | 查询关联数据，如 care 查 enrollment 获取班级信息 |
| 状态变更通知 | 事件总线 | RabbitMQ | 考勤签到、日报发布等广播事件 |
| 报表聚合 | CQRS + 只读库 | 事件驱动 | report-service 订阅所有域事件维护聚合库 |
| 文件存储 | 对象存储 | MinIO/S3 API | 成长相册、接送人照片 |

### 3. 跨库查询策略（禁止跨库 JOIN）

#### 模式 A：API 编排（实时场景）
```
家长端查看日报:

Client → API Gateway → family-service (orchestrator)
                 │
       ┌────────┼────────┐
       ↓        ↓        ↓
  care-svc  safety-svc  health-svc
  (日报)    (考勤)      (健康)
       │        │        │
       └────────┼────────┘
                ↓
          组装完整响应 → Client
```

#### 模式 B：事件驱动读模型（高频场景）
```
各服务写入自己的数据库后发布事件 →
family-service 监听事件维护本地冗余数据 →
家长查询时直接从 family-service 本地读取
```

#### 模式 C：CQRS 报表库（园长驾驶舱/监管导出）
```
各服务发布域事件 → report-service 订阅消费 →
写入 reg_dashboard / reg_compliance 等聚合表 →
园长/监管查询直接从报表库读取
```

### 4. 分布式事务：Saga 模式

复杂流程跨越多个服务时，采用编排 Saga 保证最终一致性：

```
入托流程 Saga（operation-service 编排）:

1. 创建入托档案（enr_enrollment）✓
   补偿：标记取消
2. 创建宝宝-家长绑定（baby_parent_baby）→ baby-service
   补偿：标记解绑
3. 初始化日报模板（dly_）→ care-service
   补偿：无（日报渐进生成）
4. 发送欢迎通知 → family-service
   补偿：无

全部成功 → 入托状态 = ACTIVE
任一失败 → 触发已完成步骤的补偿操作
```

### 5. ID 生成策略

| 阶段 | 方案 | 说明 |
|---|---|---|
| 当前（单体） | 数据库自增 Long | JPA `@GeneratedValue(IDENTITY)` |
| 拆分过渡期 | 统一 ID 区间分配 | 每个模块分配 ID 段 |
| 微服务期 | Snowflake ID 或 UUID v7 | 全局唯一、无需中心化协调 |

关联字段设计：只存逻辑 ID，不建物理外键约束：

```sql
-- 单体时期（可以有 FK）
enrollment_id BIGINT,
FOREIGN KEY (enrollment_id) REFERENCES enr_enrollment(id)

-- 微服务时期（只有逻辑引用）
enrollment_id BIGINT,  -- 仅存 ID，无 FK 约束
-- 查询时通过 API 或缓存获取 enrollment 详情
```

### 6. 核心事件目录

```
--- 入托生命周期 ---
enrollment.created         入托创建 → 通知家长、初始化日报、更新名额
enrollment.status.changed  入托状态变更 → 更新在托人数、通知教师
enrollment.transferred     转班 → 新旧班级教师交接

--- 考勤接送 ---
attendance.checkin         签到 → 推送家长"宝宝已到园"
attendance.checkout        签退 → 推送家长"宝宝已离园"
attendance.leave.approved  请假通过 → 更新日报、通知教师

--- 照护与日报 ---
care.record.created        照护记录 → 更新日报摘要、异常实时推送
dailyreport.published      日报发布 → 推送家长端

--- 健康安全 ---
health.observation.abnormal 晨检异常 → 推送保健医、家长、创建跟踪
incident.reported          事故上报 → 推送园长、保健医、家长确认
safety.task.overdue        安全任务逾期 → 推送后勤负责人、园长

--- 财务 ---
billing.due.reminder       账单到期提醒 → 推送家长
```

### 7. 拆分优先级

基于耦合度分析，推荐分批拆分：

```
第一批（低耦合，易拆分）
  ├── iot-gateway       ← 独立硬件协议，几乎不依赖其他服务
  └── user-service      ← 被依赖多但对外依赖极少

第二批（中等耦合）
  ├── report-service    ← 只读聚合，用事件驱动
  ├── family-service    ← 可缓存其他服务数据
  └── education-service ← 业务相对独立

第三批（高耦合，需先治理接口）
  ├── operation-service ← 入托数据被几乎所有服务依赖
  ├── care-service      ← 依赖 enrollment + attendance
  ├── health-service    ← 依赖 enrollment
  └── safety-service    ← 依赖 enrollment + classroom
```

### 8. 设计原则总表

| 原则 | 说明 |
|---|---|
| 数据独占 | 每个服务只写自己的表，其他服务只能通过 API 读 |
| 逻辑引用 | 跨服务只存 ID，不建物理外键 |
| 异步优先 | 非实时查询走事件驱动，减少同步耦合 |
| Saga 补偿 | 每个跨服务写操作必须有对应的补偿操作 |
| 先单体后拆分 | 先在代码层划清 Service 边界，微服务只是部署形态变化 |
| CQRS 仅限报表 | 不为每个查询建读模型，仅在 report-service 使用 |

后续触发条件：

- 多个团队并行开发不同业务域。
- 某个模块出现独立扩缩容需求。
- 单体发布频繁互相阻塞。
- 事件总线和消息中间件已就绪。
