# 数据模型清单

本文档记录系统所有数据模型定义。新增或修改 Entity、字段、枚举、关系时必须同步更新。

## 1. 模型总览

| Entity | 用途 | 说明 |
|---|---|---|
| `User` | 用户账号 | 实现 `UserDetails`，用于登录认证；支持 ADMIN / PARENT / ELDER 角色 |
| `Family` | 家庭 | 产品中心对象之一，聚合家庭成员关系 |
| `FamilyMember` | 家庭成员 | 用户与家庭关联，含家庭角色和权限字段 |
| `Baby` | 宝宝 | 归属于家庭，通过 Enrollment 关联机构 |
| `GrowthRecord` | 成长记录 | 照片、日记、里程碑等家庭端成长档案 |
| `Milestone` | 里程碑 | 成长里程碑记录 |
| `FamilyPost` | 家庭动态 | 家庭协作动态 |
| `FamilyTask` | 家庭任务 | 家庭成员任务 |
| `EducationPlan` | 教育计划 | 泛教育计划 |
| `EducationActivity` | 教育活动 | 计划下的具体活动 |
| `DevelopmentAssessment` | 发展评估 | 家庭/AI 场景的发育评估 |
| `AIChatSession` | AI 会话 | AI 育儿助手会话 |
| `AIChatMessage` | AI 消息 | AI 育儿助手消息 |
| `AIChat` | AI 聊天 | 需确认是否与 Session/Message 重复 |
| `BaseEntity` | 基础实体 | 通用 id、创建/更新时间抽象基类 |
| `Organization` | 托育机构 | 机构主体，支持集团关联和多园区类型 |
| `Classroom` | 托育班级 | 归属于 Organization，含托位容量和月龄范围 |
| `Staff` | 机构员工 | 归属于 Organization 并关联 User，支持多岗位角色 |
| `StaffClassroomAssignment` | 员工-班级关联 | 将教师/保育员分配到具体班级 |
| `Enrollment` | 宝宝入托档案 | 连接 Baby、Organization、Classroom，含审核状态流 |
| `AttendanceRecord` | 考勤记录 | 基于 Enrollment 记录到园、离园、缺勤、请假 |
| `LeaveRequest` | 请假申请 | 家长或园所提交，审核后生成请假考勤 |
| `CareRecord` | 一日照护记录 | 基于 Enrollment 记录喂养、饮水、睡眠、如厕、体温、情绪、活动 |
| `DailyReport` | 结构化家长日报 | 汇总考勤和照护记录，支持草稿/待审/已发布三种状态 |
| `HealthObservation` | 晨午检与全日观察 | 支持一摸二看三问四查、体温、异常和跟踪 |
| `AuthorizedPickupPerson` | 授权接送人 | 宝宝长期授权接送人 |
| `PickupDelegation` | 临时委托接送 | 临时委托接送申请、审核和接送码 |
| `AllergyTag` | 过敏标签 | 宝宝过敏源、反应、严重程度和状态 |
| `MedicationRequest` | 用药委托 | 家长提交用药申请，园所审核 |
| `MedicationAdministration` | 用药执行记录 | 实际用药时间、剂量和反应 |
| `IncidentReport` | 异常/事故上报 | 异常事故记录、处理过程、家长通知和确认 |
| `Announcement` | 通知公告 | 机构/班级通知，支持草稿和发布 |
| `AnnouncementReceipt` | 通知已读回执 | 用户已读状态记录 |
| `FeeItem` | 收费项目 | 机构收费项目定义 |
| `BillingStatement` | 收费账单 | 宝宝账单和支付状态 |
| `DirectorDashboardResponse` | 园长驾驶舱聚合 DTO | 不落库，聚合机构运营指标 |
| `AdmissionLead` | 招生线索 | 意向登记、报名审核和试托状态 |
| `MealPlan` | 食谱 | 机构每日餐次、过敏提示和营养说明 |
| `MealIntakeRecord` | 进食记录 | 宝宝实际进食量和过敏反应 |
| `SafetyLedger` | 安全卫生台账 | 消毒、食品留样、设施巡检、消防和事故跟进 |
| `ChildDevelopmentAssessment` | 儿童发展评估 | 基于入托档案的月龄里程碑和五大领域评估 |
| `RegulatoryReportResponse` | 监管报表聚合 DTO | 不落库，聚合备案、规模、人员、健康和安全卫生导出行 |
| `HardwareDevice` | 硬件设备档案 | 考勤机、晨检机器人、摄像头、门禁和电子班牌 |
| `HardwareEvent` | 硬件原始事件 | 设备事件流水，不直接替代业务记录 |
| `InfectiousDisease` | 传染病防控记录 | 疑似、确诊、隔离、康复、复园全流程 |
| `SafetyLedgerTemplate` | 安全台账周期任务模板 | 定义每天/每周/每月自动生成规则 |
| `FollowUpRecord` | 招生线索跟进记录 | 跟进内容、处理人、下次跟进时间 |
| `EnrollmentGuardian` | 入托监护人授权关系 | 链接 User 与 Enrollment，支持多种绑定方式 |
| `EnrollmentStatusHistory` | 入托状态变更历史 | 转班、暂停、复托、退托等状态变更历史 |

## 3. 第一阶段拟新增模型

### Organization

用途：托育机构主体。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | Long | 是 | 主键 |
| `name` | String | 是 | 机构名称 |
| `description` | String | 否 | 机构简介 |
| `contactPhone` | String | 否 | 联系电话 |
| `address` | String | 否 | 地址 |
| `registrationNo` | String | 否 | 监管备案编号 |
| `licenseNo` | String | 否 | 办学/托育许可证号 |
| `legalRepresentative` | String | 否 | 法定代表人 |
| `supervisorDepartment` | String | 否 | 主管部门 |
| `organizationLevel` | String | 否 | 机构等级或评定等级 |
| `operationType` | String | 否 | 运营类型 |
| `status` | Enum | 是 | ACTIVE, DISABLED |
| `orgGroupId` | Long | 否 | 所属集团品牌 ID，关联 org_group |
| `parentId` | Long | 否 | 上级机构 ID，支持连锁-分园层级 |
| `orgType` | Enum | 否 | SINGLE（单体）、HEADQUARTERS（总部）、BRANCH（分园） |
| `dailyReportApprovalRequired` | Boolean | 否 | 日报是否需要园长审核后发布，默认 false |
| `createdBy` | Long | 是 | 创建人用户 ID |
| `createdAt` | LocalDateTime | 是 | 创建时间 |
| `updatedAt` | LocalDateTime | 是 | 更新时间 |

边界：不替代 `Family`，用于机构运营和班级管理。监管字段只承接备案和导出所需的机构基础信息，不代表已完成外部监管平台直连。

### Enrollment

用途：宝宝入托关系。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | Long | 是 | 主键 |
| `babyId` | Long | 是 | 宝宝 |
| `organizationId` | Long | 是 | 机构 |
| `classroomId` | Long | 是 | 班级 |
| `status` | Enum | 是 | PENDING, HEALTH_CHECK, ACTIVE, SUSPENDED, REJECTED, WITHDRAWN |
| `enrolledAt` | LocalDate | 否 | 入托日期 |
| `allergyNotes` | String | 否 | 过敏信息 |
| `medicalNotes` | String | 否 | 健康/病史备注 |
| `specialCareNotes` | String | 否 | 特殊照护要求 |
| `emergencyContactName` | String | 否 | 紧急联系人 |
| `emergencyContactPhone` | String | 否 | 紧急联系电话 |
| `parentConfirmed` | Boolean | 是 | 家长是否已确认资料完整，默认 false |
| `parentConfirmedAt` | LocalDateTime | 否 | 家长确认时间 |
| `reviewedBy` | Long | 否 | 审核人 |
| `reviewedAt` | LocalDateTime | 否 | 审核时间 |
| `rejectReason` | String | 否 | 驳回原因 |
| `withdrawnAt` | LocalDate | 否 | 退托日期 |
| `withdrawReason` | String | 否 | 退托原因 |
| `previousClassroomId` | Long | 否 | 转班前班级 ID |
| `transferReason` | String | 否 | 转班原因 |
| `createdAt` | LocalDateTime | 是 | 创建时间 |
| `updatedAt` | LocalDateTime | 是 | 更新时间 |

状态流：`PENDING` [待园长审核] → `HEALTH_CHECK` [园长通过，待保健审核] → `ACTIVE` [保健通过，正式入托]；可被驳回至 `REJECTED`；在托后可 `SUSPENDED` 或 `WITHDRAWN`。

约束：同一机构内同一宝宝只能有一个入托档案；有效状态包含 PENDING、HEALTH_CHECK 和 ACTIVE。

### CareRecord

用途：每日保育记录。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | Long | 是 | 主键 |
| `enrollmentId` | Long | 是 | 入托档案 |
| `recordDate` | LocalDate | 是 | 日期 |
| `recordTime` | LocalDateTime | 是 | 记录时间 |
| `type` | Enum | 是 | FEEDING, WATER, SLEEP, TOILET, TEMPERATURE, MOOD, ACTIVITY |
| `valueText` | String | 否 | 文本值，例如情绪、活动名称、餐食说明 |
| `amount` | Double | 否 | 数值，例如饮水量、体温、奶量 |
| `unit` | String | 否 | 单位，例如 ml、℃、min |
| `startedAt` | LocalDateTime | 否 | 开始时间，适合睡眠、活动等持续记录 |
| `endedAt` | LocalDateTime | 否 | 结束时间，适合睡眠、活动等持续记录 |
| `remark` | String | 否 | 备注 |
| `source` | String | 是 | 数据来源，当前默认 MANUAL |
| `recordedBy` | Long | 否 | 记录人 |
| `isBackfill` | Boolean | 否 | 是否补录 |
| `backfillReason` | String | 否 | 补录原因 |
| `backfilledBy` | Long | 否 | 补录人 |
| `backfilledAt` | LocalDateTime | 否 | 补录时间 |
| `createdAt` | LocalDateTime | 是 | 创建时间 |
| `updatedAt` | LocalDateTime | 是 | 更新时间 |

边界：本模型只记录日常照护事实。记录日期早于今天或明确提供补录原因时自动标记为补录并记录补录人/时间。异常、事故、用药、晨午检、传染病等高风险数据使用独立模型，不混入普通照护记录。

## 4. 关系边界

- `Family` 负责家庭关系。
- `Organization` 负责机构关系。
- `Enrollment` 连接 `Baby`、`Organization`、`Classroom`。
- `AttendanceRecord` 和 `CareRecord` 基于入托后的宝宝记录。
- `DailyReport` 汇总考勤和保育记录，不直接替代原始记录。
- `HardwareDevice` 只描述设备归属和接入状态。
- `HardwareEvent` 只沉淀设备原始事件，不直接替代 `AttendanceRecord`、`HealthObservation`、`PickupDelegation` 等业务记录。

## 5. 新增模型前检查清单

新增 Entity 前必须确认：

- 是否已有模型可扩展
- 是否会与现有模型职责重复
- 是否需要新增枚举
- 是否需要唯一约束
- 是否需要权限校验
- 是否需要前端类型和接口
- 是否已更新本文档
