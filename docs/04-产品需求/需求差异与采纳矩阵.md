# 需求差异与采纳矩阵

本文档用于追踪客户正式需求《托育+保育平台需求（用户版-供参考）》与当前 BabyCare/HuiGrowth 路线的差异、采纳策略和任务映射。

原则：

- 客户正式需求作为产品方向和覆盖范围的主要参考。
- 当前不一次性照搬完整微服务、多端、硬件和监管方案。
- 所有核心模块都进入路线图，但按主链优先、分阶段落地。
- 若任务范围、数据模型或接口发生变化，必须同步更新本矩阵。

## 2. 模块差异明细

### 3.1 园所运营管理

客户需求重点：

- 意向登记、报名审核、资料上传、托育服务协议、试托/试园。
- 班级管理、师生比管控、班级容量、班级动态。
- 收费项目、在线缴费、退费、票据、收费统计。
- 后勤、设施报修、校车、厨房管理。

当前已有：

- `Organization` 机构模型。
- `Classroom` 班级模型。
- `Staff` 员工模型。
- `Enrollment` 入托档案模型。
- `FeeItem` 和 `BillingStatement` 已支持收费项目与账单状态。
- `DirectorDashboardResponse` 已支持园长驾驶舱基础运营指标聚合。
- `AdmissionLead` 已支持招生线索、报名审核和试托状态。

差距：

- 缺少资料上传、电子协议、招生漏斗和渠道 ROI。
- 缺少师生比、容量预警和托位使用率。
- 缺少退费、票据和在线支付。
- 缺少后勤、设施报修、厨房管理。

采纳策略：

- L1 保留最小入托链路。
- L3 增加招生/试托、收费、通知、园长驾驶舱。
- 厨房和后勤暂不作为主链，放入 L4/L5 或客户明确要求后启动。

任务映射：

- 已完成：T004、T005、T006、T007。
- 已完成：T017、T018、T019。
- 待做：招生漏斗统计、资料上传、电子协议。
- 延后：厨房管理、校车、设施报修。

### 3.2 婴幼儿照护与健康管理

客户需求重点：

- 托育端：喂养、睡眠、如厕/更换、体温、情绪、活动、辅食过敏反应。
- 保育端：进餐、午睡、饮水、如厕、户外活动。
- 健康监测：体温、心率、呼吸、血氧、身高体重、口腔检查。
- 成长档案：基础信息、发育里程碑、体检报告、作品集、成长时间轴。
- 过敏和用药管理。

当前已有：

- `Baby` 基础模型。
- `Enrollment` 已承载过敏、健康备注、特殊照护、紧急联系人。
- `CareRecord` 已承载喂养、饮水、睡眠、如厕、体温、情绪、活动等一日照护记录。
- `DailyReport` 已支持结构化日报草稿、发布和已发布日报查询。
- `AllergyTag`、`MedicationRequest`、`MedicationAdministration` 已支持过敏标签、用药委托和执行记录。
- `ChildDevelopmentAssessment` 已支持月龄里程碑和五大领域评估基础数据。
- 旧系统有成长记录、教育计划、发展评估雏形。

差距：

- 缺少体征趋势、评估图表和成长时间轴。

采纳策略：

- T009 已完成一日照护基础记录。
- T010 已完成结构化日报基础模型。
- T014 已完成用药和过敏标签基础能力。
- T022 已完成里程碑和五大领域评估基础模型，成长时间轴后续扩展。

### 3.3 卫生保健与营养膳食

客户需求重点：

- 晨午检：一摸、二看、三问、四查、全日观察。
- 食谱模板、周食谱、营养计算、带量采购、过敏替换、食谱公示。
- 传染病预警、隔离管理、密切接触者追踪、终末消毒。
- 食品采购索证、食品留样、厨房消毒、消防、设施、安全教育、事故台账。

当前已有：

- `HealthObservation` 已支持晨检、午检、全日观察和一摸二看三问四查。
- `Enrollment` 有过敏和健康备注，能作为后续保健模块输入。
- `MealPlan` 和 `MealIntakeRecord` 已支持食谱公示、过敏提示和实际进食记录。
- `SafetyLedger` 已支持消毒、食品留样、设施巡检、消防和事故跟进台账。

差距：

- 缺少保健医角色的专用工作台。
- 缺少营养计算和带量采购。
- 缺少传染病防控、营养计算和监管导出。

采纳策略：

- L2 已先做晨午检和全日观察。
- L4 已先做食谱、实际进食和安全卫生台账雏形，后续补营养计算和传染病防控。
- 硬件自动采集先不做，只预留数据来源字段。

任务映射：

- T012：晨午检与全日观察。
- T020：食谱与膳食记录雏形。已完成。
- T021：安全与卫生台账。已完成。
- T025：硬件接入抽象。

### 3.4 安全保障与安全台账

客户需求重点：

- 人脸识别接送、授权接送人、委托接送、临时接送码。
- 接送记录不可删除，仅可标记误操作。
- 滞留预警、黑名单。
- 实时监控、录像回放、AI 视频分析、明厨亮灶。
- 教职工考勤、幼儿考勤、请假审批。

当前已有：

- T008 已完成考勤与请假基础：到园、离园、缺勤、请假申请、请假审批、班级考勤和宝宝考勤查询。
- `AttendanceRecord` 已预留手工来源、体温、接送人姓名、关系和电话字段。
- `AuthorizedPickupPerson` 和 `PickupDelegation` 已支持授权接送人与临时委托接送。
- `IncidentReport` 已支持异常事故上报、处理、关闭和家长确认。
- `SafetyLedger` 已支持安全卫生台账基础能力。
- 尚未实现硬件接入。

差距：

- 缺少滞留预警。
- 缺少硬件、监控和黑名单能力。

采纳策略：

- T008 已做到园、离园、请假、缺勤和班级考勤。
- T013 已完成接送授权和委托接送基础能力。
- T015 已完成异常/事故上报基础能力。
- T025 已完成设备档案和原始事件抽象，不做真实硬件接入。

### 3.5 家园共育与隔代共育

客户需求重点：

- 班级动态、一日报告、通知公告、成长相册、发展评估。
- 体征趋势、晨午检记录、食谱查看、用药查看、请假、接送通知、委托接送。
- 长辈模式：大字号、高对比、语音播报、简化入口、接送码、家庭群组。

当前已有：

- 家庭、家庭动态、家庭任务、成长记录等历史能力。
- `familyStore` 已统一到真实 `/family/*` API。
- `ParentReports` 已支持家长查看已发布日报、考勤和照护记录。
- `Announcement` 和 `AnnouncementReceipt` 已支持机构/班级通知与已读回执。

差距：

- 当前家园能力仍偏家庭协作，不是机构班级消息流。
- 祖辈模式已有入口雏形，但语音播报、家庭群组和更完整通知聚合仍需继续增强。

采纳策略：

- T010/T011 已跑通家长日报和查看。
- T016 已完成通知公告和已读回执基础能力。
- T023 已完成祖辈模式入口雏形，复用日报、考勤和接送真实 API。
- 家庭动态后续应升级为班级动态和宝宝动态，不再作为泛社区。

### 3.6 课程教学与发展评估

客户需求重点：

- 托育端 6-36 月龄发展里程碑。
- 保育端 3-6 岁五大领域评估。
- 发展雷达图、班级发展分布、学期发展增量。
- 个别化教育计划 IEP。

当前已有：

- 后端已有 `DevelopmentAssessment`、`EducationPlan`、`EducationActivity` 等旧模型。
- `ChildDevelopmentAssessment` 已基于 `Enrollment` 承接机构入托后的专业评估。
- 前端教育规划页面目前仍是骨架。

差距：

- 旧模型没有按正式需求区分托育月龄里程碑和保育五大领域；T022 已用新模型补上基础结构。
- 缺少过程性观察记录和前端图表展示。
- 缺少与一日照护、活动记录的联动。

采纳策略：

- T022 已重新梳理评估模型，不急于扩展现有教育计划页面。
- 已先做数据结构和评估结果，复杂图表后续补。

### 3.7 数据统计与卫健监管对接

客户需求重点：

- 园长驾驶舱：运营、健康、安全、教学、家园、财务。
- 监管字段：机构、班级、人员、在托婴幼儿、健康保健、收托规模。
- 等级评定：安全、健康、膳食、照护、家园、人员、财务、数据上报。

当前已有：

- Dashboard 已重构为托育工作台骨架。
- 机构、班级、员工、入托基础数据已经开始沉淀。
- 园长驾驶舱聚合接口已支持机构、班级、在托、出勤、异常、账单和通知基础指标。

差距：

- Dashboard 页面仍未全面图表化，当前先具备真实统计 API。
- 监管字段与导出行已有雏形，但还缺少地区监管口径、Excel/CSV 文件生成和等级评定规则。
- 缺少等级评定指标。

采纳策略：

- T018 已完成园长驾驶舱真实数据聚合接口。
- T024 已完成监管字段和导出行雏形，不做外部系统直连。
- 等级评定先做指标清单和数据覆盖，不做自动评级复杂规则。

### 3.8 多角色功能矩阵

客户需求重点：

- 园长、教师、保健医、家长、祖辈均有不同权限和终端。
- 同一功能在不同角色下查看/操作权限不同。

当前已有：

- `Staff` 有 DIRECTOR、TEACHER、CAREGIVER、FINANCE。
- `User` 仍只有 PARENT、ADMIN。

差距：

- 缺少 HEALTH_DOCTOR、SECURITY、GRANDPARENT、PLATFORM_ADMIN 等角色。
- 缺少真正的 RBAC、菜单权限和数据权限。
- 前端还未形成多角色路由和工作台。

采纳策略：

- 短期先在 `Staff` 侧补机构员工角色。
- 家长/祖辈关系从 FamilyMember 和接送授权中扩展。
- 后续做单 Web 多角色工作台，不立即拆多端。

## 3. 暂缓或不直接照搬的内容

| 客户需求 | 当前处理 | 原因 |
|---|---|---|
| 微服务、API Gateway、K8s、灰度发布 | 暂缓，采用模块化单体 | 当前业务主链未闭环，过早拆分会增加复杂度 |
| Flutter App、微信小程序、监管平台并行 | 暂缓，采用单 Web 多角色 | API 和业务边界仍在变化，多端并行会造成返工 |
| 人脸识别、AI 视频、晨检机器人、IoT 网关 | 只预留字段和接口抽象 | 当前没有真实设备和协议文档 |
| 真实卫健/教育/市场监管接口直连 | 先做字段和导出 | 地区差异大，需要明确客户和接口文档 |
| OCR、电子签、在线支付、电子发票 | 后续园所运营阶段再做 | 主链闭环优先级更高 |

## 4. 后续维护规则

- 若新增任务来自客户正式需求，必须先在本矩阵中定位所属模块。
- 若决定暂缓某项客户需求，必须记录暂缓原因。
- 若实现某项能力，必须更新对应任务 ID 和状态。
- 若客户需求发生变更，先更新本矩阵，再更新任务板和数据模型。
