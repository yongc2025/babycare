# 托育 + 保育产品规划

## 1. 文档目的

本文档记录 BabyCare/HuiGrowth 从“家庭育儿记录型产品”升级为“真实托育 + 保育平台”的产品方向、能力边界和阶段路线。

2026-07-15 已补充用户正式需求《托育+保育平台需求（用户版-供参考）》的评估结论。本项目不照搬其完整微服务、多端、硬件和监管方案，但吸收其中对托幼一体、合规、安全、健康、祖辈端和园所运营的优秀设计。

## 2. 当前产品策略

### 架构策略

当前不拆微服务，采用“模块化单体”。

```text
一个 Spring Boot 后端
一个 React 前端
按业务模块划清边界
为未来微服务和多端拆分预留接口
```

详细决策见 `docs/02-架构设计/架构决策.md`。

### 多端策略

当前不同时建设 PC、App、小程序、监管端。

先用一个 React Web 承载多角色工作台：

- 园长 / 机构管理员
- 老师 / 保育员
- 保健医
- 家长
- 财务
- 后勤 / 保安

当前已实现：
- **祖辈模式**（`ElderMode`）已上线：大字高对比界面、只读查看日报和接送状态。
- **老板多园区驾驶舱**（`BossDashboard`）已上线：跨园区运营指标聚合。

后续在主链稳定后，再拆教师移动端、家长小程序和监管端。

### 产品优先级

第一优先级不是硬件、AI、微服务或监管对接，而是跑通真实托育主链：

```text
机构 -> 班级 -> 入托档案 -> 考勤接送 -> 一日照护 -> 日报 -> 家长同步 -> 园所数据沉淀
```

当前状态：主链已基本跑通，L1-L2 阶段功能完整可用，L3-L4 已进入收尾，L5 已启动部分能力。

## 3. 正式需求带来的校准

用户正式需求比原规划更完整，覆盖：

- 园所运营：招生线索、报名审核、试托、分班、收费账单、退费、后勤。
- 婴幼儿照护：喂养、睡眠、如厕、饮水、体征、活动、成长档案、发展评估。
- 卫生保健：晨午检、食谱营养、营养膳食分析、传染病防控、食品留样、消毒台账。
- 安全保障：接送授权、委托接送、考勤记录、事故记录、安全台账、安全周期任务。
- 家园共育：日报、动态（家庭圈）、通知公告、家庭任务、家长申请。
- 隔代共育：长辈模式（大字高对比界面已上线）、接送码。
- 课程评估：0-3 岁月龄里程碑、3-6 岁五大领域评估、教育计划、教学活动。
- 数据监管：园长驾驶舱、老板驾驶舱、备案字段、等级评定、监管报表。
- 硬件部署：考勤机、晨检机器人、摄像头、电子班牌、环境检测（设备与事件模型已抽象）。

我们接受其产品方向，分阶段落地，当前 L1-L4 已基本完成。

## 4. 双龄段产品架构

正式需求中“托育端”和“保育端”的差异非常关键，后续模型和页面需要支持机构或班级级别配置。

| 维度 | 托育端 0-3 岁 | 保育端 3-6 岁 |
|---|---|---|
| 关注重点 | 精细照护、健康监测、安全防护 | 教学评估、家园互动、幼小衔接 |
| 一日流程 | 喂养、睡眠、换尿布、抚触、体征 | 进餐、午睡、活动、课程、户外 |
| 健康管理 | 发育曲线、过敏、喂养、每日体征 | 晨午检、体质测评、视力口腔筛查 |
| 评估体系 | 月龄发展里程碑 | 健康、语言、社会、科学、艺术 |
| 家园沟通 | 高频结构化日报 | 日精选、周报、月报、作品展示 |

短期先按托育主链实现，数据模型预留 `careType` / `classroomType` / `ageRange` 这类字段。

## 5. 角色与端规划

### 园长 / 机构管理员 / 老板

- 机构、园区、班级、员工管理
- 集团/品牌管理（多园区架构）
- 招生线索、报名审核、试托、入托、分班
- 师生比、班级容量、托位使用率
- 审批、通知公告、数据驾驶舱（园长驾驶舱 + 老板多园区驾驶舱）
- 合规、台账、监管导出
- 入托审核、转班、退托、复托全流程

### 老师 / 保育员

- 今日班级宝宝列表、照护任务清单
- 到园、离园、请假、缺勤
- 晨检、全日观察
- 喂养、饮水、睡眠、如厕、体温、情绪、活动
- 照片、视频、教师评语
- 异常事件、用药执行、日报生成与发布审核

### 保健医

- 晨午检管理
- 体征异常处理
- 用药审核与执行
- 过敏标签管理
- 食谱营养与营养膳食分析
- 传染病防控（疑似/确诊/隔离/康复/复园全流程）
- 卫生保健台账导出

### 家长

- 查看到离园状态
- 查看日报、动态（家庭圈）、照片、通知公告
- 请假申请
- 委托接送
- 用药申请
- 入托关系绑定、资料补全、确认
- 手机号注册 / 短信验证
- 邀请码绑定宝宝和入托档案
- 查看成长档案、体征趋势、账单
- 微信登录预留

### 祖辈家长

- 大字、高对比、少入口（`ElderMode` 已上线）
- 看孩子、看日报、看接送状态
- 接送码 / 实体卡
- 只读权限隔离

### 财务 / 运营

- 收费项目管理
- 账单、缴费、欠费、退款
- 收据 / 发票
- 财务报表
- 招生漏斗与跟进提醒

### 后勤 / 保安

- 接送核验
- 安全台账与周期任务
- 过期提醒
- 设施报修
- 校车 / 门禁 / 监控协作

### 监管人员

当前不建独立监管平台，已支持：
- 监管报表导出
- 备案字段预留
- 等级评定基础数据

### 系统管理员

- RBAC 权限模型（角色、权限、菜单）
- 用户管理、角色分配
- 审计日志
- 系统配置、数据字典

## 7. 当前不优先做

以下能力有价值，但在主链和真实客户场景全面跑通后再投入：

- 物理微服务拆分
- K8s、灰度、服务网关、链路追踪
- 原生 App / 小程序多端并行开发
- 真实监管系统直连对接
- 人脸识别、AI 视频分析
- OCR、电子签、在线支付、电子发票
- 大规模 AI 包装和泛内容社区

## 8. 当前状态总结

截至 2026-07-19，托育主链（L1-L3）已基本跑通，L4 专业保健与教学评估接近完成，L5 多端与硬件已启动部分能力。

后续开发重点：
- L4 收尾：食品留样、消毒台账
- L5 推进：AI 日报草稿、教师移动端/PWA、家长小程序
- 全量测试与质量加固：当前 120 个测试用例 113 通过

## 9. 结论

正式用户需求证明：真正的托育 + 保育平台不是简单记录宝宝成长，而是支撑园所完成“运营、照护、健康、安全、家园、监管”的完整闭环。

我们的路线应保持务实：

```text
产品上接受完整平台视野
架构上采用模块化单体
前端上采用单 Web 多角色
实现上优先跑通主链
```

这样既能贴近用户的正式需求，又能避免过早陷入微服务、多端和硬件集成的复杂度。
