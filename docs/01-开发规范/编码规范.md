# 编码规范与开发边界

本文档约束后续所有开发，避免文件过长、函数过大、重复造轮子和业务边界失控。

## 1. 通用原则

- 优先复用现有架构、工具和命名方式。
- 后端遵循 Controller -> Service -> Repository -> Entity 分层。
- 前端遵循 Page -> Component -> Service -> Store 的边界。
- 新增功能必须有真实数据流，不允许只做静态页面。
- 新增表、字段、接口、核心函数必须同步记录到 docs。
- 所有源代码文件（`.java`、`.ts`、`.tsx`、`.css`、`.json`、`.xml`、`.properties`、`.sql` 等）和注释**必须使用 UTF-8 编码**，禁止 GBK、GB2312、ISO-8859-1 等其他编码。
- 如果发现源码或文档乱码，必须先修复乱码（转为 UTF-8），再继续功能开发。
- 判断中文内容是否真实乱码时，必须优先用 `node`/`rg` 验证文件真实内容，不得只依据 PowerShell `Get-Content` 的显示结果。
- 单次任务只完成一个明确业务闭环，避免大面积无关 diff。
- 前端开发必须使用全局统一风格，不允许另外编写样式。
- 本系统是windows开发环境，生成命令时必须考虑windows可执行。
- 开始编码前必须先检查项目里有没有类似的模式？能不能复用/扩展已有的逻辑？

### 1.1 抽象设计与开发效率

- 系统开发必须有抽象化设计，但抽象必须服务于真实业务边界、稳定复用点或明确复杂度，不允许为了行数、命名统一或假想未来需求制造空壳 helper、空壳 service、空壳 util。
- 新增能力前必须先做 1-2 分钟设计判断，并在实现中遵循优先级：复用已有 -> 扩展现有 -> 新建。只有已有模块语义不匹配、扩展会破坏边界，或出现明确新业务边界时，才允许新建模块/类型/工具。
- 优先拆真实业务边界：按领域职责、数据所有权、权限边界、状态流转边界拆分；不得只因为文件接近行数上限就机械拆出没有独立语义的类或函数。
- 重复逻辑需要沉淀复用：同一模式出现 3 次以上，且语义稳定、调用方收益明确时，应提取公共 helper、mapper、组件、hook 或 util；不足 3 次但风险高、容易写错的逻辑也可以提前沉淀。
- 批量读取：开始改动前一次性读取相关文件的大范围内容，并用 `rg` 并行搜索 Entity、Repository、Service、Controller、前端 API、类型、页面和文档引用，避免边改边猜。
- 批量编辑：同一业务闭环内的多文件修改应集中完成，尽量一次性处理同层同类问题，避免每改一个小文件就反复停顿。
- 批量验证：完成一整层或一个业务闭环后再统一构建/测试；不要每改一个小文件就完整验证。高风险公共逻辑除外，可先做局部快速验证。
- 本节是“抽象、复用、效率”的主定义；其他流程文档只做执行承接，不重复定义相同原则。

## 2. 文件长度限制

### 后端

| 文件类型 | 建议上限 | 硬上限 | 处理方式 |
|---|---:|---:|---|
| Entity | 220 行 | 300 行 | 拆枚举、值对象或关联对象 |
| DTO | 180 行 | 250 行 | 拆 Request/Response |
| Controller | 220 行 | 300 行 | 按资源拆 Controller |
| Service | 350 行 | 500 行 | 拆领域 Service 或私有 helper |
| Repository | 120 行 | 180 行 | 保持查询职责单一 |
| Config | 180 行 | 250 行 | 拆配置类 |

### 前端

| 文件类型 | 建议上限 | 硬上限 | 处理方式 |
|---|---:|---:|---|
| Page | 350 行 | 500 行 | 拆子组件、hooks、constants |
| Component | 220 行 | 320 行 | 拆展示组件 |
| Store | 220 行 | 320 行 | 拆模块 store |
| Service/API | 260 行 | 380 行 | 按业务模块拆 service |
| Types | 250 行 | 400 行 | 按领域拆 types |
| CSS | 300 行 | 450 行 | 拆页面样式 |

超过建议上限时必须评估拆分；超过硬上限不得继续堆代码，优先拆真实业务边界，不为行数制造空壳 helper。

## 3. 函数长度限制

| 类型 | 建议上限 | 硬上限 |
|---|---:|---:|
| Controller 方法 | 40 行 | 60 行 |
| Service 公共方法 | 80 行 | 120 行 |
| Service 私有方法 | 60 行 | 90 行 |
| React 事件处理函数 | 50 行 | 80 行 |
| React 组件 render 主体 | 180 行 | 260 行 |
| 数据转换函数 | 50 行 | 80 行 |

超出时优先拆为校验函数、数据转换函数、私有业务步骤、子组件或自定义 hook。拆分必须有清晰语义，优先拆真实业务边界，不为行数制造空壳 helper。

## 4. 后端规范

### 包结构

```text
com.huigrowth.babycare
├── config
├── controller
├── dto
├── entity
├── exception
├── repository
├── security
├── service
└── util
```

新增模块优先沿用该结构，不新建平行架构。

### 命名规范

- Entity：名词单数，例如 `Organization`、`Classroom`
- Repository：`XxxRepository`
- Service：`XxxService`
- Controller：`XxxController`
- Request DTO：`XxxCreateRequest`、`XxxUpdateRequest`
- Response DTO：`XxxResponse`
- 枚举：优先放在 Entity 内部；如被多个模块复用，再抽为独立文件

### Controller 边界

Controller 只做：

- 接收请求
- 读取认证用户
- 参数校验注解
- 调用 Service
- 包装 `ApiResponse`

Controller 不做：

- 复杂查询
- 权限业务判断
- 数据拼装
- 状态流转

### Service 边界

Service 负责：

- 业务校验
- 权限校验
- 状态流转
- 调用 Repository
- Entity 与 DTO 转换

Service 不负责：

- HTTP 细节
- 前端展示逻辑
- 原始 SQL 拼接，除非 Repository 已无法表达

### Repository 边界

Repository 负责：

- JPA 查询
- 命名查询方法
- 必要的 `@Query`

Repository 不负责：

- 权限判断
- 业务状态修改
- DTO 组装

## 5. 数据库规范

| 域 | 前缀 | 未来微服务 | 说明 |
|---|---|---|---|

| 机构 | `org_` | operation-service | 机构、班级、员工、园区关系 |
| 入托 | `enr_` | operation-service | 入托档案、入托状态变更 |
| 考勤接送 | `att_` | safety-service | 考勤记录、请假申请、授权接送人、委托接送 |

| 日报 | `dly_` | care-service | 家长日报、AI 草稿 |
| 健康 | `hlth_` | health-service | 晨午检、用药委托、用药执行、过敏标签、事故上报 |
| 食谱 | `meal_` | health-service | 食谱计划、进食记录 |
| 安全 | `safety_` | health-service | 安全卫生台账、巡检任务 |
| 公告 | `ann_` | family-service | 通知公告、已读回执 |
| 财务 | `fin_` | operation-service | 收费项目、账单、支付、退款 |
| 招生 | `adm_` | operation-service | 招生线索、跟进记录、转化记录 |
| 家庭 | `family_` | family-service | 家庭、家庭成员、家庭动态、家庭任务 |
| 宝宝 | `baby_` | care-service | 宝宝信息、家长绑定、邀请码 |

| AI | `ai_` | care-service | AI 会话、AI 消息 |

#### 表名规则

```
格式：{域前缀}_{实体英文名_snake_case}

示例：
org_organization
org_classroom
att_attendance_record
hlth_health_observation
sys_role
sys_audit_log
```

| 规则项 | 要求 |
|---|---|
| 命名风格 | 全小写 snake_case |
| 单复数 | 使用单数（`org_classroom` 而非 `org_classrooms`） |
| 分隔符 | 下划线 |
| 域前缀 | 每个表必须有所属域的前缀 |
| 关联表 | `{域}_{主体}_{关联}`，如 `org_staff_classroom` |

#### 列名规则

```
格式：snake_case

示例：
organization_id
check_in_time
created_at
is_active
attendance_date
```

| 类型 | 规则 | 示例 |
|---|---|---|
| 主键 | `id` | `id` |
| 外键 | `{关联表单数}_id` | `organization_id`, `classroom_id` |
| 日期 | `{语义}_date` | `attendance_date`, `report_date` |
| 时间戳 | `{语义}_at` | `created_at`, `checked_in_at`, `published_at` |
| 布尔 | `is_{形容词}` | `is_active`, `is_abnormal` |
| 布尔 | `{过去分词}_{名词}` | `follow_up_required`, `parent_notified` |
| 枚举 | 直接名词 | `status`, `type`, `severity`, `scope` |
| 数值 | `{语义}` 或 `{语义}_{单位}` | `amount`, `temperature`, `capacity` |
| 文本 | `{语义}` | `remark`, `description`, `reason` |
| JSON | `{语义}_json` | `meta_json`, `allergen_json` |

#### Java 字段 → 数据库列映射

```java
// Java (camelCase)          DB (snake_case)
organizationId         →    organization_id
checkInTime            →    check_in_time
createdAt              →    created_at
isActive               →    is_active
followUpRequired       →    follow_up_required
parentNotified         →    parent_notified
```

#### 表名 ↔ Entity 名对应

```text
数据库表（snake_case）        Java Entity（PascalCase）
org_organization            →  Organization
org_classroom               →  Classroom
att_attendance_record       →  AttendanceRecord
hlth_health_observation     →  HealthObservation
sys_role                    →  SysRole 或 Role（类名不含前缀）
```

#### 索引命名

| 索引类型 | 格式 | 示例 |
|---|---|---|
| 主键 | `PRIMARY`（默认） | — |
| 唯一索引 | `uk_{表名}_{字段}` | `uk_sys_user_username` |
| 普通索引 | `idx_{表名}_{字段}` | `idx_att_attendance_enrollment_date` |
| 联合索引 | `idx_{表名}_{f1}_{f2}` | `idx_att_attendance_org_date` |
| 外键 | `fk_{子表}_{父表}` | `fk_att_attendance_enrollment` |

#### Java → MySQL 类型映射

| Java 类型 | MySQL 类型 | 说明 |
|---|---|---|
| `Long` | `BIGINT` | 主键、外键 |
| `String`（短） | `VARCHAR(20)` | 枚举值、状态码 |
| `String`（中） | `VARCHAR(255)` | 名称、电话、邮箱 |
| `String`（长） | `TEXT` | 备注、内容、描述 |
| `LocalDate` | `DATE` | 日期（无需时分秒） |
| `LocalDateTime` | `DATETIME` | 时间戳 |
| `Boolean` | `TINYINT(1)` | 布尔值（0/1） |
| `BigDecimal` | `DECIMAL(10,2)` | 金额 |
| `Double` | `DOUBLE` | 数值度量（体温、体重等） |
| `Integer` | `INT` | 整数（容量、月龄等） |
| `Enum` | `VARCHAR(20)` | `@Enumerated(STRING)` |

### 5.2 BaseEntity 规范

所有业务实体默认继承 `BaseEntity`：

```java
@MappedSuperclass
public class BaseEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(updatable = false)
    private LocalDateTime createdAt;

    private LocalDateTime updatedAt;
}
```

按需选择性继承以下混入：

| 混入接口 | 额外字段 | 适用场景 |
|---|---|---|
| `SoftDelete` | `deletedAt` | 需软删除的表（如用户、入托档案） |
| `TenantAware` | `organizationId` | 多租户数据隔离（如所有机构相关表） |
| `Auditable` | `createdBy`, `updatedBy` | 需操作审计的表 |
| `Versioned` | `version` | 并发更新场景（乐观锁） |

### 5.3 约束要求

字段必须明确：

- 是否必填（`nullable = false`）
- 是否唯一（`unique = true`）
- 最大长度（`length = xxx`）
- 默认值（`columnDefinition` 或 Java 字段初始值）
- 是否可修改（`updatable = false` 用于创建后不可变字段）

### 5.4 枚举字段

枚举必须使用：

```java
@Enumerated(EnumType.STRING)
```

不得使用 ordinal。枚举值使用全大写 snake_case：

```java
public enum AttendanceStatus {
    ABSENT, CHECKED_IN, CHECKED_OUT, LEAVE
}
```

### 5.5 关联关系

默认使用显式外键字段或 `ManyToOne`，避免滥用双向 `OneToMany`。

新增关联必须说明：

- 拥有方
- 删除策略
- 查询入口
- 权限边界

跨模块（不同域前缀的表）只存逻辑 ID，不建物理外键约束：

```java
// 当前单体阶段：可以有 @ManyToOne
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "enrollment_id")
private Enrollment enrollment;

// 未来微服务阶段：只存 ID，通过 API 查询
@Column(name = "enrollment_id")
private Long enrollmentId;
```

### 5.6 域数据边界规则

```
┌──────────────────────────────────────────────┐
│ 每个域的表只能由所属微服务（或模块）直接写入     │
│ 其他域通过 API 读取，不跨域直连 JOIN            │
│ 跨域引用只存 ID，不建物理外键约束               │
│ 共享基础表（sys_user）所有域可读，仅 user 模块可写 │
└──────────────────────────────────────────────┘
```

当前模块化单体阶段可在代码层通过 Service 层调用访问其他模块的数据，但新增跨模块查询场景必须先记录到 `docs/02-架构设计/API设计.md`，注明当前为跨模块调用，为未来拆分预留接口边界。

## 6. 前端规范

### API 调用

所有 HTTP 请求必须通过：

```text
frontend/src/services/api.ts
```

或后续按模块拆出的 service 文件。

不得在页面中直接 `fetch`，除非是明确迁移任务中的临时清理点。

### 页面职责

Page 负责：

- 页面布局
- 调用 hooks/store/service
- 组合组件
- 处理页面级状态

Page 不负责：

- 大量业务规则
- 大量数据转换
- 复杂表单子逻辑
- 散落拼接 API URL

### Store 职责

Store 负责跨页面共享状态，例如当前用户、当前机构、当前班级。

Store 不负责：

- 大量接口方法集合
- 复杂表单状态
- 页面局部状态

### Mock 数据规则

- 新功能不得新增 mock fallback。
- 允许在测试文件或明确命名的 `mock` 文件中保留测试数据。
- 页面接口失败时必须展示错误或空状态，不得展示假数据。
- 暂未接入真实接口的页面必须标注为“待接入”状态。

## 7. 交付文档更新

每轮任务交付时必更新的文档清单和验收要求以 `docs/01-开发规范/AI工作流.md` §7 为唯一真源。

新增或变更 API/数据模型时，具体需更新的内容按以下清单执行（交付时统一更新）：

**涉及 `docs/02-架构设计/API设计.md`：**

- Controller 路由
- 前端 API service 方法
- 关键 Service 公共方法
- Store action

**涉及 `docs/02-架构设计/数据模型.md`：**

- Entity
- 字段
- 枚举
- 表关系
- 重要索引或唯一约束

## 8. 重复造轮子检查

新增代码前必须按“复用已有 -> 扩展现有 -> 新建”的顺序检查：

- 是否已有类似 Entity
- 是否已有 Repository 查询
- 是否已有 Service 方法
- 是否已有 API service 方法
- 是否已有前端类型定义
- 是否已有通用组件
- 是否已有错误处理方式

如果复用困难，先记录原因，再新增。新增后如果形成稳定重复模式，必须按 1.1 节沉淀为可复用能力。

## 9. 验收规范

每个任务的验收标准必须包含：

- 后端接口是否可调用
- 前端页面是否接真实数据
- 空状态是否可见
- 错误状态是否可见
- 权限是否校验
- 文档是否更新
- 构建或测试是否通过，或说明未验证原因
