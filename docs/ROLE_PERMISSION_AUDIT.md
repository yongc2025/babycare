# 角色工作内容与权限边界审计

最近更新：2026-07-16

## 1. 审计结论

当前系统已经具备托育保育主链的业务对象和页面骨架，但角色体系仍处在雏形阶段。

当前代码实际角色：

| 层级 | 现有角色 | 说明 |
|---|---|---|
| 登录用户角色 | `ADMIN`, `PARENT` | `User.UserRole` 只有管理员和家长 |
| 机构员工岗位 | `DIRECTOR`, `TEACHER`, `CAREGIVER`, `FINANCE` | `Staff.StaffRole` 只用于机构员工身份 |
| 家庭成员角色 | `CREATOR`, `PARENT`, `GRANDPARENT`, `RELATIVE` | 用于家庭内部协作，不等同于系统登录权限 |

主要问题：

- `ADMIN` 同时承担系统管理员、老板/机构管理员、园长、教师、保健、安全、财务等多种职责，边界过宽。
- `Staff` 岗位还没有真正参与菜单、接口、按钮和数据权限控制。
- 多园区/集团统筹模型不足，当前 `Organization` 更像单个园区，不适合作为集团主体。
- 入托档案、健康安全、财务、招生、监管等责任边界需要进一步产品化。
- 缺少“系统管理后台”“老板多园区驾驶舱”“园长园区管理台”“保健安全工作台”等清晰入口。

后续原则：

```text
登录角色决定系统身份
员工岗位决定机构内职责
组织关系决定数据范围
页面按钮决定操作边界
审计日志记录关键责任
```

## 2. 建议角色模型

### 2.1 平台/系统管理员

定位：维护系统运行、账号、权限和基础配置，不参与具体园区日常业务。

工作内容：

- 初始化系统配置、品牌信息、数据字典。
- 创建平台管理员或重置关键账号。
- 管理全局角色、菜单、按钮权限。
- 查看所有机构/园区基础状态。
- 配置短信、文件上传、通知、登录策略。
- 查看审计日志、异常日志和数据健康状态。
- 处理数据修复、字符集、脏数据、账号冻结等系统问题。

权限边界：

- 可查看全平台配置和机构清单。
- 可创建/禁用平台账号。
- 原则上不直接修改宝宝日常照护记录、家长日报、收费明细，除非进入数据维护流程。

责任边界：

- 对系统可用性、权限配置、数据安全负责。
- 不对单个园区运营结果、宝宝照护过程负责。

当前已有：

- 开发环境自动初始化 `admin/admin123`。
- 配置文件支持 CORS、数据库、上传目录、JWT 等。

当前缺少：

- 系统设置页面。
- 平台账号管理。
- 角色权限配置。
- 数据字典管理。
- 审计日志。
- 全局机构/园区管理。

关联任务：

- T048 系统管理后台规划与入口。
- T049 RBAC 角色权限模型。
- T050 审计日志与关键操作留痕。

### 2.2 老板/机构管理员

定位：机构主体或集团负责人，关注经营结果、风险、资源和多园区统筹。

工作内容：

- 创建和维护机构主体信息。
- 创建多个园区。
- 给每个园区分配园长。
- 查看多园区经营数据。
- 对比园区托位、出勤、招生、收费、健康、安全、师生比。
- 查看异常预警和监管材料完整度。

需要看的内容：

- 园区数、班级数、总托位、已用托位、空余托位。
- 今日出勤率、请假率、异常健康记录。
- 本月新增入托、本月退托、招生线索转化。
- 应收、已收、欠费、退费。
- 安全事故、未关闭台账、监管缺失字段。
- 各园区排名和趋势对比。

权限边界：

- 可看自己机构主体下所有园区。
- 可创建园区、任命园长、查看多园区汇总。
- 不应直接替教师修改每日照护过程，除非具备审计原因。

责任边界：

- 对经营、合规、资源配置、园区负责人任命负责。
- 不对班级每日具体记录的真实性承担第一责任。

当前已有：

- `Organization` 可创建机构/园区。
- `DirectorDashboard`、`OperationsRegulatory` 已有单机构维度聚合。

当前缺少：

- 集团/机构主体模型。
- 多园区关系。
- 园区负责人任命。
- 多园区驾驶舱。
- 园区对比、排名、风险预警。

关联任务：

- T051 多园区组织模型设计。
- T052 老板/机构管理员多园区驾驶舱。
- T053 园区负责人任命与数据范围控制。

### 2.3 园长

定位：单个园区实际负责人，对本园区运营、人员、班级、入托、风险和监管负责。

工作内容：

- 维护本园区资料和监管备案字段。
- 创建班级和托位容量。
- 创建/邀请员工。
- 分配员工岗位和班级。
- 审核宝宝入托、退托、转班。
- 查看本园区出勤、日报、健康、安全、收费、招生、监管。
- 处理未完成日报、异常健康、安全事故、欠费、满班等风险。

权限边界：

- 可管理本园区机构资料、班级、员工、入托档案。
- 可查看本园区全部宝宝和业务记录。
- 可审核关键流程，但不应替老师批量伪造日常照护记录。
- 不可查看其他园区数据，除非同时拥有老板/机构管理员身份。

责任边界：

- 对本园区运营结果、人员配置、班级容量、入托审核、异常闭环负责。
- 对教师和保育员的每日记录进行监督，不是每条记录的第一记录人。

当前已有：

- `/organization-management` 支持机构、班级、员工、入托管理。
- `/dashboard` 和 `/operations-regulatory` 可看部分运营监管数据。
- `StaffRole.DIRECTOR` 已存在。

当前缺少：

- 园长专属登录/菜单权限。
- 园长与具体园区的强绑定。
- 班级员工分配。
- 入托审核、转班、退托流程。
- 园长待办和风险预警。

关联任务：

- T054 园长工作台与园区数据范围。
- T055 班级员工分配。
- T056 入托审核、转班、退托流程。

### 2.4 教师

定位：班级教育与家园沟通负责人，负责班级日常观察、教学发展、日报内容。

工作内容：

- 查看自己负责班级和宝宝。
- 记录到园、离园、请假辅助处理。
- 记录活动、情绪、发展观察。
- 编辑并发布家长日报。
- 查看家长反馈、通知回执。
- 参与成长评估、教育计划。

权限边界：

- 只能看自己负责班级或被授权宝宝。
- 可创建本班考勤、照护、健康观察、日报、发展评估。
- 不应创建园区、修改财务、修改监管备案、任命员工。

责任边界：

- 对班级记录、日报内容、教育发展观察负责。
- 对食品安全、财务收款、园区合规不承担主责。

当前已有：

- `/teacher-workbench` 已有班级照护页。
- `/daily-report-management` 已有日报管理。
- `StaffRole.TEACHER` 已存在。

当前缺少：

- 教师账号与班级授权。
- 教师专属菜单。
- 日报发布审核策略。
- 教师和保育员在同一记录上的分工。

关联任务：

- T057 教师班级授权与菜单边界。
- T058 日报发布审核策略。

### 2.5 保育员

定位：宝宝一日生活照护第一记录人。

工作内容：

- 记录喂养、饮水、睡眠、如厕、体温、情绪、活动。
- 配合晨午检和全日观察。
- 发现异常时上报教师/保健员。
- 查看本班今日任务和未完成照护记录。

权限边界：

- 只能操作本班照护记录。
- 可新增和补充照护记录。
- 不应发布日报、修改机构资料、处理财务、关闭安全事故。

责任边界：

- 对照护记录的及时性和真实性负责。
- 对健康诊断、监管报表、收费不承担主责。

当前已有：

- `CareRecord`、`TeacherWorkbench` 已支持一日照护记录。
- `StaffRole.CAREGIVER` 已存在。

当前缺少：

- 保育员专属工作台。
- 保育员班级授权。
- 照护任务清单和漏记提醒。
- 记录补录审核。

关联任务：

- T059 保育员工作台与照护任务清单。
- T060 照护补录与审核机制。

### 2.6 保健员/保健医

定位：园区卫生保健、健康异常、用药、传染病防控和营养膳食负责人。

工作内容：

- 查看晨午检、全日观察和异常记录。
- 处理健康异常和跟进记录。
- 管理过敏、用药委托、用药执行。
- 管理传染病上报、隔离、复园。
- 管理食谱、营养分析、食品留样。
- 输出卫生保健台账和监管材料。

权限边界：

- 可查看本园区健康、用药、过敏、食谱和相关台账。
- 可处理健康异常和用药审核。
- 不应修改财务、招生、机构经营数据。

责任边界：

- 对健康安全专业判断、用药执行、食品卫生、传染病防控负责。
- 不对招生转化、收费、日常教学评价负责。

当前已有：

- `/health-safety` 已聚合晨午检、用药、事故、台账。
- `HealthObservation`、`MedicationRequest`、`MealPlan`、`SafetyLedger` 已有基础模型。

当前缺少：

- `HEALTH_WORKER` / `HEALTH_DOCTOR` 员工角色。
- 保健员专属菜单和工作台。
- 用药审核责任人。
- 传染病防控流程。
- 营养分析和保健统计。

关联任务：

- T061 扩展保健员角色。
- T062 保健工作台。
- T063 传染病防控流程。
- T064 营养膳食分析。

### 2.7 安全/后勤负责人

定位：设施安全、消防、消毒、食品留样、环境卫生、隐患闭环负责人。

工作内容：

- 新增和维护安全卫生台账。
- 跟进隐患、事故、设施巡检。
- 维护消毒、食品留样、消防检查记录。
- 关闭处理完成的台账。

权限边界：

- 可操作安全卫生台账和事故跟进。
- 可查看关联事故与园区安全统计。
- 不应操作宝宝日报、财务账单、招生线索。

责任边界：

- 对安全卫生记录和隐患闭环负责。
- 对医疗专业判断、教学记录不承担主责。

当前已有：

- `SafetyLedger` 和 `/health-safety` 台账入口。
- `IncidentReport` 基础模型。

当前缺少：

- `SAFETY_OFFICER` / `LOGISTICS` 员工角色。
- 后勤安全工作台。
- 台账周期任务、过期提醒。
- 隐患整改闭环责任人。

关联任务：

- T065 扩展安全/后勤角色。
- T066 安全台账周期任务与过期提醒。

### 2.8 财务/运营

定位：收费、账单、招生线索、试托转化、经营效率负责人。

工作内容：

- 配置收费项目。
- 创建账单、标记支付、取消账单。
- 管理招生线索、报名审核、试托、转入托。
- 查看欠费、收款、招生转化、托位使用率。

权限边界：

- 可操作收费、账单、招生线索。
- 可查看必要的宝宝和入托信息。
- 不应修改健康诊断、日常照护原始记录、机构系统配置。

责任边界：

- 对收费准确性、招生推进、经营数据负责。
- 对宝宝日常照护真实性不承担主责。

当前已有：

- `StaffRole.FINANCE` 已存在。
- `Billing`、`AdmissionLead`、`OperationsRegulatory` 已有基础能力。

当前缺少：

- 运营/招生岗位。
- 财务专属菜单与权限。
- 收费审核、退费、发票、支付对接。
- 招生漏斗、转化率、跟进提醒。

关联任务：

- T067 扩展运营/招生角色。
- T068 财务运营工作台。
- T069 招生漏斗与跟进提醒。

### 2.9 家长

定位：宝宝监护人，查看宝宝在园情况并与园区协同。

注册与激活策略：

- 家长可以通过公开注册入口创建账号，但公开注册只能创建 `PARENT`。
- 家长注册建议以手机号为主，用户名为兼容字段，后续逐步弱化用户名。
- 正式环境应启用短信验证码；当前开发阶段可以先保留密码注册。
- 微信登录/微信手机号授权适合作为后续增强，不作为第一阶段唯一入口。
- 园长或招生/运营可以在后台生成邀请链接/邀请码，用于把家长绑定到指定宝宝或入托档案。
- 没有园区邀请时，家长只能创建家庭空间和宝宝基础资料，不能看到园区、班级和入托数据。

工作内容：

- 查看宝宝日报、考勤、照护、健康观察。
- 查看通知公告并确认。
- 管理接送人和委托接送。
- 提交请假、用药委托。
- 查看账单和缴费状态。
- 与园区进行家园沟通。

权限边界：

- 只能查看自己家庭/宝宝相关数据。
- 可提交家长侧申请和确认。
- 不可查看其他宝宝、其他班级、园区经营数据。

责任边界：

- 对家庭资料、接送授权、请假、用药委托真实性负责。
- 不对园区照护记录的创建负责。

当前已有：

- `PARENT` 登录角色。
- 家庭管理、家长日报、长辈模式、家园协作等入口。
- `FamilyMember` 有家庭角色。

当前缺少：

- 家长与入托档案之间的正式绑定流程。
- 家长手机号注册、短信验证、邀请绑定策略。
- 家长提交请假/用药/接送的完整页面闭环。
- 账单查看和支付。
- 家长消息沟通闭环。

关联任务：

- T070 家长入托关系绑定。
- T071 家长申请闭环增强。
- T072 家长账单查看与支付预留。

### 2.10 长辈/其他监护人

定位：低操作门槛的信息查看者。

工作内容：

- 大字版查看日报摘要、到离园、吃睡健康状态。
- 查看接送提醒和重要通知。
- 可根据授权确认部分通知。

权限边界：

- 默认只读。
- 是否允许接送确认、请假、用药，应由家长或园区授权。
- 不应操作园区管理、财务、日报发布。

责任边界：

- 对本人确认的接送/通知行为负责。
- 不对宝宝资料维护和园区记录负责。

当前已有：

- `/elder-mode` 长辈模式页面。
- `FamilyMember.FamilyRole.GRANDPARENT` 已存在。

当前缺少：

- 长辈登录身份与家长授权。
- 长辈可操作范围配置。
- 语音播报和通知确认细化。

关联任务：

- T073 长辈授权与只读权限。

## 3. 入托责任边界

宝宝入托不是单一动作，建议拆成流程：

```text
招生/运营录入线索
-> 家长补充宝宝和监护人资料
-> 园长审核托位、班级和入园日期
-> 保健员补充健康、过敏、特殊照护信息
-> 园长确认入托生效
-> 老师/保育员开始日常记录
```

### 3.1 家长注册与宝宝关联决策

家长账号、宝宝资料和入托档案应分开设计：

```text
User 家长账号
-> Family 家庭
-> Baby 宝宝基础资料
-> Enrollment 入托档案
-> Organization/Classroom 园区和班级
```

推荐注册方式：

| 场景 | 注册/激活方式 | 说明 |
|---|---|---|
| 家长主动了解系统 | 手机号 + 短信验证码注册 | 只创建家长账号和家庭空间，不自动绑定园区 |
| 园区已录入宝宝 | 园长/招生生成邀请链接或邀请码 | 家长注册后通过邀请码绑定宝宝和入托档案 |
| 家长先提交报名 | 手机号注册后填写报名/宝宝资料 | 进入招生线索，由园区审核后转入托 |
| 微信生态使用 | 微信登录/手机号授权 | 后续增强，适合家长端和长辈端 |
| 员工账号 | 后台创建或邀请激活 | 不允许公开注册成员工 |

第一阶段建议：

```text
家长公开注册：手机号/用户名 + 密码，后续补短信验证码
园区绑定：园长/招生在后台生成邀请码
宝宝信息：可以由园长建档，也可以由家长补充，最终由园长确认入托
```

正式产品建议：

```text
手机号作为主账号
短信验证码用于注册、找回密码和关键绑定
微信登录作为便捷登录
邀请码用于园区、宝宝、入托档案的安全绑定
```

宝宝信息创建责任：

| 信息类型 | 推荐创建/维护角色 | 说明 |
|---|---|---|
| 宝宝姓名、性别、生日 | 家长或园长/招生 | 家长填写更准确，园区可代录 |
| 监护人、联系方式 | 家长 | 园区可审核 |
| 紧急联系人、接送人 | 家长 | 园长/老师只审核和查看 |
| 过敏、病史、特殊照护 | 家长填写，保健员审核 | 涉及健康风险，需专业角色确认 |
| 入园日期、班级、状态 | 园长 | 属于园区管理责任 |
| 招生线索、试托记录 | 招生/运营 | 园长最终确认入托 |

绑定关系建议：

```text
园长/招生创建或确认 Enrollment
-> 系统生成 inviteCode / invitationLink
-> 家长用手机号注册或登录
-> 输入邀请码
-> 系统创建 FamilyMember 和 EnrollmentGuardian 绑定
-> 家长获得该宝宝的日报、考勤、照护、请假、接送、用药等权限
```

安全边界：

- 家长不能通过搜索宝宝姓名直接绑定宝宝。
- 邀请码应有有效期、使用次数和绑定范围。
- 一个宝宝可绑定多个监护人，但至少一个主监护人。
- 老师只能查看本班宝宝和已授权家长联系方式。
- 长辈默认由家长邀请，权限低于主监护人。

简化版本：

```text
园长创建入托档案
-> 分配班级
-> 绑定家长
-> 录入紧急联系人和健康备注
-> 生效
```

当前系统：
