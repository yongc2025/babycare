## 4. 缺陷清单

### BUG-001：昵称中文编码显示异常

- **严重级别**：Medium
- **分类**：DATA
- **角色**：ADMIN / PARENT
- **页面/路径**：全局（侧栏、用户按钮、日报）
- **前置条件**：注册时填写中文昵称
- **复现步骤**：
  1. 注册用户时 nickname 为中文，例如"管理员T040"
  2. 登录后在任何显示昵称的位置查看
- **期望结果**：显示"管理员T040"
- **实际结果**：显示"???T040"（问号代替中文字符，英文字母正常）
- **接口信息**：
  - URL: POST /api/auth/register
  - Response: `"nickname": "???T040"`（中文字符全部变为问号）
- **初步判断**：MySQL 连接字符集或表字符集未正确设置 utf8mb4，中文字符被截断。后端日志无报错，说明数据存储层字符集问题。

---

### BUG-002：入托档案创建后入园日期显示"未填写"

- **严重级别**：Low
- **分类**：DATA
- **角色**：ADMIN
- **页面/路径**：园所运营 → 入托档案 → 档案列表
- **前置条件**：创建入托档案时填写了入园日期
- **复现步骤**：
  1. 创建入托档案时在"入园日期"字段选择日期
  2. 保存后在档案列表查看"入园日期"列
- **期望结果**：显示已填写的入园日期
- **实际结果**：显示"未填写"
- **初步判断**：前端列表展示字段与创建/编辑表单字段可能不一致，或者后端返回时未正确填充该字段。

---

### BUG-003：注册接口角色固定为 PARENT

- **严重级别**：Low
- **分类**：PRODUCT
- **角色**：未知
- **页面/路径**：POST /api/auth/register
- **前置条件**：调用注册接口
- **复现步骤**：
  1. 调用注册接口，不传 role 字段
  2. 查看生成的用户角色
- **期望结果**：可通过参数指定角色，或根据场景自动判断
- **实际结果**：后端 `AuthService.register()` 始终设置 `user.setRole(User.UserRole.PARENT)`
- **代码位置**：`backend/.../AuthService.java`，约第 62 行
- **初步判断**：产品设计决策——注册只能创建 PARENT 角色。如需 ADMIN 需要手动改 DB，测试中已通过 Python 脚本绕过。

---

### BUG-004：注册接口手机号空字符串未校验

- **严重级别**：Low
- **分类**：API
- **角色**：未知
- **页面/路径**：POST /api/auth/register
- **前置条件**：前端发送 `phone: ""`（空字符串）
- **复现步骤**：
  1. 调用注册接口，phone 传空字符串 `""`
  2. 后端检查 `request.getPhone() != null` 通过（空字符串 ≠ null）
  3. 后续唯一性约束使用空字符串查询/写入
- **期望结果**：空字符串应被视为"未填写"或明确校验
- **实际结果**：空字符串通过 null 检查，但后续可能导致唯一性约束冲突
- **代码位置**：`backend/.../AuthService.java`，约第 62 行
- **初步判断**：需要在注册前对空字符串做 `StringUtils.hasText()` 校验。

### BUG-005：TeacherWorkbench 和 HealthSafety 页面缺乏创建按钮

- **严重级别**：Low
- **分类**：UX
- **角色**：ADMIN / TEACHER
- **页面/路径**：今日工作台 / 健康安全
- **前置条件**：需要有对应权限
- **复现步骤**：
  1. 进入"今日工作台"页面
  2. 查看健康观察和照护记录区域
  3. 进入"健康安全"页面查看用药/事故/台账区域

- **实际结果**：所有页面均为纯展示，没有任何创建入口
- **初步判断**：这些功能可能依赖于特定角色（如 TEACHER）或特定状态才能显示创建按钮，或者产品设计上尚未实现。测试中通过 API 直接调用创建。

### BUG-006：CORS 配置未包含端口 3001

- **严重级别**：Info
- **分类**：API
- **角色**：未知
- **页面/路径**：backend application-dev.properties
- **前置条件**：前端从 localhost:3001 发起请求
- **复现步骤**：
  1. 直接从前端（localhost:3001）访问后端 API（无 Vite 代理）
- **期望结果**：允许跨域请求
- **实际结果**：`application-dev.properties` 中 `cors.allowed-origins=http://localhost:3000,http://127.0.0.1:3000` 未包含 3001
- **影响**：当前通过 Vite 代理（`/api` → localhost:8080）工作正常，但直接访问会被 CORS 阻止。生产环境应该通过反向代理解决，但 dev 配置建议更新。
